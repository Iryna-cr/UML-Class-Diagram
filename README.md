# UML-Class-Diagram

## Опис предметної області:

Система бронювання квитків у кіно створена для зручного вибору фільму, дати, сеансу та місця у залі. Користувач може переглядати доступні фільми, бронювати або купувати квитки онлайн. Адміністратор системи має можливість додавати нові фільми, створювати сеанси та контролювати продаж квитків. Система зберігає інформацію про користувачів, їхні замовлення та статуси бронювання для подальшої обробки.

## Class Diagram

### На діаграмі класів зображені основні сутності системи:

Користувач - реєструється, входить у систему, створює бронювання;

Бронювання - містить інформацію про дату, статус і пов’язані квитки;

Квиток - містить дані про місце, ряд, статус та належить певному сеансу;

Сеанс - зберігає дані про час, зал і фільм;

Фільм - має назву, жанр, тривалість;

Платіж - відповідає за оплату бронювання.


### Між класами встановлено такі зв’язки:

Користувач - Бронювання, асоціація: один користувач може створювати кілька бронювань (наприклад, квитки на різні фільми або дати), але кожне конкретне бронювання належить лише одному користувачу. Це асоціація, бо між ними є логічний зв’язок, але об’єкти існують незалежно — користувач може існувати без бронювання, і навпаки (якщо бронювання зробив інший користувач);

Бронювання - Квиток, композиція: одне бронювання може містити декілька квитків (наприклад, якщо користувач бере кілька місць поруч). Квитки не можуть існувати без бронювання, адже вони створюються саме в процесі бронювання;

Бронювання - Оплата, асоціація: кожне бронювання має один відповідний платіж, що відображає його оплату. Але оплата і бронювання - це окремі об’єкти, які можуть існувати незалежно (наприклад, оплату ще не здійснено);

Сеанс — Квиток, агрегація: один сеанс фільму може мати багато квитків (на різні місця). Квиток пов’язаний із сеансом, але може бути створений або видалений незалежно від самого сеансу (наприклад, якщо квиток скасували);

Фільм — Сеанс, агрегація: один фільм може мати декілька сеансів (у різні дні, години або зали). Якщо фільм видаляється із системи (наприклад, закінчився прокат), його сеанси зазвичай теж видаляються, але технічно можуть існувати тимчасово незалежно.


### Архітектурний підхід:

Система реалізована за принципом шарової архітектури, що поділяє проєкт на три основні рівні:

1)Інтерфейс користувача - забезпечує взаємодію користувача із системою через вебінтерфейс;

2)Бізнес-логіка - відповідає за обробку даних, перевірку правил бронювання та оплату;

3)Доступ до даних - зберігає інформацію про фільми, сеанси, користувачів і квитки в базі даних.

Такий підхід забезпечує розділення відповідальності, спрощує супровід і масштабування системи.
